Database Internals

How my data is stored.

A live peek inside the two SQLite databases I use most: my personal RAG (Karpathy-style knowledge base) and the Workflowy MCP cache. Tap any row to expand.

TL;DR

Both databases are SQLite — single files on disk. Nothing is stored as separate Markdown or HTML files. All the content lives as TEXT values inside columns of rows inside tables.

RAG stores plain Markdown-style text in a content column, plus a metadata column holding JSON-as-text.

Workflowy stores HTML in name and note columns (because Workflowy supports inline bold, links, tags), plus a parent_id pointer that builds the tree.

Both ship with a parallel FTS5 index — a hidden mirror table that makes search instant.

FTS5 · how search is instant

SQLite ships with a built-in full-text search engine called FTS5. The RAG creates a "shadow" virtual table called documents_fts that mirrors the content + title columns of documents, but is indexed for word-level search.

Three triggers (documents_ai, documents_au, documents_ad) keep the shadow table in sync — every insert / update / delete on documents automatically updates the index.

End result: /rag-query "no-code automation" returns matching docs in ~1ms, not minutes. No vector embeddings required.

How the tree gets built · parent_id pointers

There's no nested JSON. The hierarchy lives in the parent_id column. Each row points at the row above it. Reconstructing the tree means querying recursively (or just looping in code).

-- The query that builds that tree:

WHERE parent_id = '0003ae82-3654-9984-177f-ee39ac5f84d4'
ORDER BY priority;

Why the FTS5 index matters here too

Searching 228K nodes for the word "flywheel" by scanning every row would take seconds. nodes_fts indexes every word in name + note, so wf_find_node "flywheel" returns in milliseconds.

This is the same pattern as the RAG: a virtual FTS5 table mirrors the content of the real table, kept in sync by triggers, queried with a special MATCH operator instead of LIKE.

Takeaways

SQLite is just one file. Both databases are a single .db file on your hard drive. Easy to back up, easy to inspect with any SQLite tool, easy to query with Python in 3 lines.

Text columns hold all the meaningful content. No "documents" in the file-system sense. The data is rows in tables.

Markdown vs HTML is a stylistic choice, not a database choice. The RAG ingests sources that are mostly plain prose, so Markdown-style text round-trips cleanly. Workflowy ships HTML because that's what its UI emits — bold/italic/links survive the round-trip.

Search speed comes from FTS5. Without it, querying 219K + 228K rows on every search would be too slow. With it, every query is essentially instant.

Relationships make it a graph, not a list. The RAG's relationships table and Workflowy's parent_id are doing the same kind of job — encoding structure beyond what the doc text alone can hold.
