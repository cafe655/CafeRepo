Architectural Framework · v0.1 Outline

How to build a website you can actually run.

A 9-layer framework for solo non-developers building real production websites with an AI coding agent. Each layer pre-answers questions so neither you nor Claude has to re-decide them per page.

May 1, 2026 · For David (Cafe) + Claude
The Big Idea

The framework is organized as layers. Each layer pre-answers questions for the layer above it, so neither you nor the agent has to re-decide them per page or per project.

The lower layers are read more often (by you, by Claude, on every change). The upper layers are read when you're making decisions or recovering history.

When operational, the contract is: Claude reads layers 1–6 before writing any code, and writes to layers 7–9 as a side effect of its work.

The Layers
Layer 1
The Foundational Mental Model
The principles that govern every other layer.

Intent

Before any tool choice, the project needs a small set of operating principles that the rest of the framework can be checked against. These principles travel with you across projects.

Core principles

  • Decisions, not options. A good architecture narrows choice. One type ramp, one spacing scale, one component for "card." Every "or" is a future inconsistency.
  • Tokens, not values. Components reference named tokens (--bg-page, space-4), never raw values. (W3C DTCG spec, Primer's foundational rule.)
  • Schema-first contracts. Every shape — content, props, API responses, DB rows — has one canonical schema. Zod is the de-facto standard. The schema is the contract, the validator, and the type.
  • Constraint as feature. Limited palettes, limited components, limited prose styles → consistency for free. Agents pick randomly when given options; humans get tired when given options.
  • Separation of concerns by layer. Content ≠ presentation. Presentation ≠ layout. Layout ≠ structure.
  • Explicit over implicit. Especially for AI agents: a written-down rule is followed, an inferred rule drifts.
  • Reversible defaults. Pick patterns that can be replaced incrementally, not patterns that demand rewrites.
  • Vibe-coding requires guardrails. "Talk to Claude and push" only stays sustainable if the framework catches drift automatically.
When fully developed: a 1-page principles statement that other layers cite.
Layer 2
The Specification Layer
What we're building, before we build it.

Intent

A spec is the artifact that gets written before code. It captures the "what" and the "why" so the agent doesn't have to invent answers from gaps in the prompt. Two dominant frameworks have emerged in 2025–2026: GitHub Spec Kit (Constitution → Specify → Clarify → Plan → Tasks → Implement) and Amazon's Kiro (Requirements → Design → Tasks).

Components

  • specs/ directory at repo root, one folder per feature.
  • spec.md — what is being built, success criteria, out of scope. Acceptance criteria in EARS notation: "When [trigger], the [system] shall [behavior]."
  • plan.md — how we'll build it. Architectural choices, file changes, sequencing.
  • tasks.md — ordered, parallelizable work items.
  • Lifecycle: spec → human review → plan → human review → implementation. Not optional, even for small features.
  • Solo-dev lightweight version: for trivial changes, a single spec.md is enough. For anything touching multiple layers, do the full flow.
Why this matters for AI-augmented dev: Without a spec, the agent fills gaps with guesses; with a spec, the agent fills gaps with the spec. This is the single highest-leverage discipline change for solo vibe-coders.

Sources

  • GitHub Spec Kit
  • Kiro IDE
  • Martin Fowler: Spec-Driven Development
  • PaulDuvall/ai-development-patterns
When fully developed: spec.md / plan.md / tasks.md templates and a one-page "when to write a spec" decision rule.
Layer 3 · highest leverage
The Agent Operating Manual
How Claude reads the project on every turn.

Intent

AGENTS.md is the cross-tool spec (Linux Foundation, supported by Claude Code, OpenAI Codex, Cursor, Aider, Copilot, etc.) for briefing AI agents on a codebase. CLAUDE.md is the Claude-specific extension; many projects symlink them. Vercel does. The agent reads the nearest file in the directory tree, so subprojects can override the root.

Components of a good AGENTS.md

  • Project overview (what & why, 2–3 lines)
  • Stack & versions (Next.js 16, Tailwind v4, Supabase, Vercel)
  • Codebase map (where things live)
  • Build/dev/test commands (exact strings to run)
  • Code style — only what tools can't enforce
  • Testing instructions
  • Security / secrets boundaries
  • Commit / PR conventions
  • Known gotchas / anti-patterns
  • Pointers to deeper docs (progressive disclosure)

Hard rules from the field

  • Brevity is load-bearing. Frontier LLMs reliably follow ~150–200 instructions; Claude Code's system prompt already uses ~50. Target under 300 lines, ideally under 60. Every line costs context budget on every turn.
  • Don't auto-generate. Hand-craft. Auto-gen produces noise.
  • Progressive disclosure. Root file thin, deeper docs (docs/auth.md, docs/db-conventions.md) loaded on demand.
  • Forbidden patterns belong here. Explicit Don't list (no inline style=, no Tailwind arbitrary values, no any).

Sources

  • agents.md (Linux Foundation spec)
  • vercel/next.js AGENTS.md (gold standard)
  • HumanLayer: Writing a good CLAUDE.md
When fully developed: a template AGENTS.md plus a checklist for what not to put in it.
Layer 4
The Design System
Tokens, components, patterns — pre-decides every visual question.

Intent

Without a design system, every page reinvents type, color, spacing, motion. With one, the agent (and you) reach for the same tokens every time, and the site looks coherent without anyone having to think about it.

4.1 Tokens

  • Two-layer token model: primitive (gray.1–gray.12, space.4) → semantic (bg.page, fg.muted, border.default). Components reference only semantic tokens.
  • DTCG-compliant JSON as source of truth. W3C Design Tokens spec reached first stable version Oct 2025.
  • Build pipeline: Style Dictionary emits CSS variables, Tailwind theme blocks, Figma styles from the same JSON.
  • Sub-site theming: each sub-site overrides only the semantic mapping (its own bg.emphasis, font.sans), never the components.

4.2 Foundations chapters

Color · Typography · Spacing · Radius · Elevation · Motion · Breakpoints · Iconography. Each chapter: rules, allowed values, do/don't, code examples.

4.3 Components, patterns, accessibility, voice

Components (Button, Card, Input, Pill, Nav, Footer, Badge, Dialog, Toast). Patterns (page shell, hero, article layout, empty/loading/error states, forms). Accessibility baked into tokens (Radix's 12-step scales pre-encode AA targets). Voice & content style: title case rules, sentence length, link text, error messages — same status as visual rules.

4.4 Tailwind v4 integration

Tailwind v4 dropped JS config; tokens now live in CSS via @theme. Pipeline: design.tokens.json → Style Dictionary → theme.css. Sub-site overrides via [data-subsite="foo"] selector.

4.5 AI-agent ergonomics

  • Self-documenting names (--fgColor-onEmphasis) beat short names.
  • Explicit Do/Don't blocks adjacent to the tokens.
  • Decision rules over examples.
  • One canonical answer per question. No "pick one" lists.
  • Forbidden list: no inline style=, no arbitrary Tailwind values, no new colors without adding a token first.

Canonical references to study

  • Radix Colors — cleanest semantic-step model
  • GitHub Primer + primer/primitives
  • Vercel Geist
  • Tailwind v4 theme docs
  • W3C Design Tokens Format Module
  • Style Dictionary
  • Google's DESIGN.md — Markdown-first, JSON-embedded
When fully developed: full ToC, a starter design.tokens.json (primitive + semantic), the Tailwind v4 wiring, and per-component spec sheets.
Layer 5
The Content Model
What content is, where it lives, who edits it.

Intent

Right now, content on cafe655.com is mashed into static HTML files in public/ — content and presentation fused. The content layer separates what the page says from how the page looks, so editing copy doesn't require touching code, and changing design doesn't risk corrupting copy.

5.1 Content type taxonomy

  • Sub-sites — name, slug, tagline, theme, hero image, enabled flag
  • Cards / pill-links — title, href, icon, parent sub-site, order
  • Long-form pages — title, slug, MDX body
  • Site settings — nav order, sub-site enable/disable
  • Dynamic data — form submissions, comments, analytics — Postgres territory

5.2 Where content lives — the decision tree

Content typeHomeWhy
Long-form proseMDX in content/Human-readable, diff-friendly, embeds React
Structured listsJSON/YAML in content/Schema-validatable, fast to render
Edited via UICMS collection (Keystatic)UI on top of the same MD/JSON files
Dynamic dataSupabase (Postgres)Real DB for changing-without-deploy data

5.3 Schema-first contracts

Every content type has a Zod schema. The schema is:

  • The runtime validator (parse on read, fail fast)
  • The static type (z.infer<typeof CardSchema>)
  • The CMS collection definition (Keystatic generates forms from it)
  • The DB schema (Drizzle ORM derives Zod schemas from Supabase tables)

One canonical definition. Edit it once, every consumer updates.

5.4 Migration path from static HTML

The public/*.html pages are the current liability. Migration approach:

  • For each HTML page, extract the prose to MDX with frontmatter.
  • Move the inline styles into the design system.
  • Drop the MDX into the appropriate sub-site's content/ folder.
  • Delete the static HTML.

This is where Claude earns its pay — paste an HTML file, get back MDX + missing-token list.

Sources

  • Zod docs
  • Drizzle ORM + drizzle-zod

Layer 6
The Code Architecture
How files are organized so the agent always knows where things go.

Intent

A predictable folder structure means new features always land in the same kind of place, the agent's "where do I put this?" answer is deterministic, and a stranger reading the repo can navigate it.

6.1 Hub-and-spoke layout

src/
├── app/
│   ├── layout.tsx          # ROOT — only place with <html><body>
│   ├── page.tsx            # Hub landing
│   ├── (sites)/            # Route group — sub-sites here
│   │   ├── _template/      # Copy this to start a new sub-site
│   │   ├── ai-field-notes/
│   │   │   ├── layout.tsx
│   │   │   ├── page.tsx
│   │   │   ├── _components/
│   │   │   ├── _lib/
│   │   │   └── [slug]/
│   │   └── …
│   ├── admin/              # Layer 7
│   └── api/
├── components/{layout,ui,seo}/
├── lib/{schemas,content,db,utils}/
└── styles/tokens/
content/                    # Layer 5
specs/                      # Layer 2
docs/decisions/             # Layer 9

6.2 The _template/ pattern

A real sub-site starter: layout.tsx, page.tsx, opengraph-image.tsx, _components/SiteShell.tsx, _lib/content.ts, and a README with a 5-line rename checklist. Copying this folder is how a new sub-site is born.

6.3 Server vs. client components

  • Server by default. Layouts, pages, navbar.
  • Push 'use client' to leaves. A nav with a mobile-menu toggle stays a server component; only the toggle is client.
  • Don't import server components from client components. Use children slot.
  • server-only package on data-fetchers to prevent client leaks.

6.4 Subdomain routing

Modern Next.js 16: proxy.ts (renamed from middleware.ts) at app root, rewrites *.cafe655.com to /[subsite]/.... Vercel needs the wildcard domain configured; Cloudflare needs a wildcard CNAME.

6.5 Sub-site theming via CSS variables

Each sub-site's layout.tsx sets data-subsite="ai-field-notes". CSS overrides target that selector. Components stay theme-blind.

6.6 Next.js 16 specific gotchas

  • middleware.ts is renamed proxy.ts. Codemod available. Your repo still has middleware.ts.
  • params and searchParams are now Promises — always await.
  • cacheComponents flag replaces ppr / useCache / dynamicIO.
  • Bundled docs at node_modules/next/dist/docs/ — version-matched, agent should prefer these.

Sources

  • Next.js 16 docs
  • Project structure guide
  • Vercel Platforms Starter Kit

Layer 7
The Admin Interface
How you edit the site without writing code.

Intent

The friend's "click a heading and edit it" experience is the destination. The framework picks a path that keeps the codebase tiny, AI-friendly, and progressively earns capability.

7.1 Admin UX taxonomy

PatternUXCost
Separate dashboardLog in, sidebar, edit in formLow
In-place editingClick a heading on the live siteMed-high
Inspector panelRight-rail with component tree + propsHigh
Visual builderFigma-like canvas (Builder.io, Plasmic)High lock-in
Git-based CMSUI writes MD/JSON; edits = commitsLow-med
DB-backed CMSSchema-defined Studio (Sanity, Payload)Medium

7.2 Phased adoption — recommended for cafe655.com

Phase 1 · Minimum viable admin (1–2 sessions)
Supabase magic-link auth → /admin route → list of sub-sites → forms backed by a site_content table (text, image URLs, links, nav order, enable/disable). Public site reads server-side.
Phase 2 · In-place editing
?edit=1 query mode renders the public site with edit overlays. <Editable field="hero.title" /> wraps editable strings, opens a popover, PATCHes Supabase. ~200 lines of custom code.
Phase 3 · Inspector / visual builder
At this point evaluate Keystatic (git-based, content-as-files, free) or Sanity + Vercel Visual Editing (hosted, content source maps, zero component instrumentation). Both are cheaper than building Phase 3 by hand.

7.3 Auth recipe — solo magic link

  • Supabase project, email magic link enabled, allow-listed email.
  • /login calls supabase.auth.signInWithOtp({email}).
  • proxy.ts calls getUser() (never getSession() server-side — Supabase security note), redirects unauth'd /admin/* to /login.
  • Optional: second check inside /admin layout that user's email matches allow-list.

7.4 Admin-as-component-library principle

The admin UI imports the same Hero, Card, Heading components as the public site. Form-driven props on real components. Storybook is the development-time version; in production it becomes /admin/preview.

Layer 8
The Operational Loop
How code goes from your prompt to the live site, and how you verify it.

Intent

Without a tight feedback loop, AI-generated changes rot silently. The loop is the safety net that lets vibe-coding scale without breaking things.

The loop

Edit → typecheck → lint → test → commit → push → preview → eyeball → merge. Every step automated except the eyeball.

Components

  • Pre-commit hooks (Husky + lint-staged): Prettier, ESLint, TypeScript strict, Zod schema parse on changed schemas. Blocks bad commits before push.
  • CI gates on every PR: pnpm typecheck && pnpm lint && pnpm test && pnpm build. Single command must pass.
  • Vercel preview deploys on every push: unique URL per branch. Eyeball on real infrastructure, not localhost.
  • Visual regression via Playwright snapshots (or Chromatic): screenshots of every key page diffed against baseline. Agent can't merge if screenshots changed unexpectedly.
  • Rollback path: vercel rollback, or revert commit. Documented so it's not surprising under pressure.
  • The ?edit=1 preview mode (Layer 7) is the manual companion: click around in admin to preview changes before publishing.

Sources

  • Autonoma: regression testing Vercel previews
  • Visual regression with Playwright + GitHub Actions
When fully developed: the package.json scripts, the .husky/ setup, the GitHub Actions workflow, and the Playwright config.
Layer 9
The Decision Record
Memory of why you chose this, not just what.

Intent

ADRs (Architecture Decision Records) and the newer AgDR (Agent Decision Records) capture the why of architectural choices. Without them, the agent re-invents the design from scratch every conversation, and you lose the rationale that justified each decision.

Components

  • docs/decisions/ folder. One Markdown file per decision: 0001-keystatic-over-sanity.md, 0002-supabase-rls-pattern.md.
  • Y-statement format: "In the context of [situation], facing [concern], we chose [decision] to achieve [goal], accepting [tradeoff]."
  • Metadata header (AgDR style): date, agent, model, trigger, status (accepted / superseded / deprecated).
  • AGENTS.md rule: the agent reads the latest 5 ADRs before any architectural change, and writes a new ADR when introducing a new pattern.
  • Pair with session log: the log captures what we did, ADRs capture why we chose this.

Sources

  • adr.github.io
  • me2resh/agent-decision-record (AgDR)
  • Adolfi.dev: AI-generated ADRs
When fully developed: the ADR template, an "is this ADR-worthy?" decision rule, and the AGENTS.md snippet that wires it into the agent loop.

What's Next

This is the outline. Reading order suggestion:

1. Read it once end-to-end. Notice which sections feel right, which feel wrong, which feel missing.

2. Push back. Sections you'd add, sections you'd cut, sections you'd reorder, principles you disagree with.

3. We pick a layer to fully develop first. My recommendation: Layer 3 (Agent Operating Manual) — highest leverage, lowest cost. Once your AGENTS.md is right, every subsequent layer's quality goes up.

After you sign off on the outline shape, each layer expands into its own working document, and the layers begin to wire into the live cafe655.com codebase one at a time.
