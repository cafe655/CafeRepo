The Big Idea

The framework is organized as eleven layers. Each layer pre-answers questions for the layer above it, so neither you nor the agent has to re-decide them per page or per project.

The contract: Claude reads layers 1–7 before writing any code. Layer 4 (Security) is non-negotiable — every other decision is checked against it. Layer 9 (Version Control) is the substrate every change runs on. Layers 8 (Admin) and 10 (Operational Loop) are the day-to-day surfaces. Layer 11 (Decision Record) is written as a side effect.

The framework is good iff it pays back its own cost — the metric is time from "Cafe has an idea" to "live in production" gets shorter, not longer.

The Layers
Layer 1 · 14 principles
The Foundational Mental Model
Constitutional law for the codebase. Every other layer cites these.

"Every application has an inherent amount of irreducible complexity. The only question is who deals with it." — Larry Tesler.

Read these as constitutional law: where complexity lives, who pays the cost of choice, what the agent is allowed to invent versus what it must look up.

1.1 Decisions, not options.

Every "or" in your spec is a future inconsistency. Tesler's Law: complexity cannot be eliminated, only moved. The architect's job is to absorb it once so the implementer never has to. Linear's "one really good way of doing things" is the reference.

Violates when: Two Card components in the same week. Three valid border-radius values. Six months later nobody knows which is "right."

1.2 Tokens, not values.

Components reference named tokens, never raw values. For humans this is hygiene. For AI agents it is survival — Claude pattern-matches on names. Tokens make the right choice the only choice the pattern-matcher can find.

Violates when: A new section ships with style={{ background: '#FAFAFA' }}. Three months later the brand color shifts and that section silently doesn't.

1.3 Schema-first contracts.

Every shape has exactly one canonical Zod schema — runtime validator, TypeScript type, contract. One file, one truth, three uses. The single highest-leverage discipline for AI-augmented dev.

Violates when: A ContactSubmission interface in types.ts, a contactSchema in lib/validation.ts, and a contact_submissions table whose columns drift independently.

1.4 Constraint as feature.

Limited palettes/components/styles produce better outcomes than infinite flexibility. Linear's strategic constraint is what allows it to feel inevitable. For an AI codebase: Claude generates the most coherent code when the search space is narrow.

Violates when: "We might need this someday" components — <FlexibleHero> with eight variants, six never used.

1.5 Separation of concerns by layer.

Content ≠ presentation ≠ layout ≠ behavior. Each lives in a different home. Mixing them is the single largest source of "this is hard to change" debt.

Violates when: A landing-page hero with hardcoded h1, inline color, custom fetch, and grid-template all in one file. Now everything about that page is a code change.

1.6 Explicit over implicit.

Every rule the agent should follow is written down somewhere it will read. AGENTS.md exists precisely for this — projects with detailed AGENTS.md files average 35–55% fewer agent-generated bugs. Defends against Hyrum's Law: if you don't write down which behaviors are intentional, all become load-bearing.

Violates when: "We always use kebab-case" — a rule no one wrote down, which Claude obeys 80% of the time.

1.7 Reversible defaults.

Pick patterns whose lock-in cost is low. Keystatic over Sanity (content portable as MD/JSON). Supabase over Firebase (Postgres is portable). Markdown over proprietary block editor. Vercel — Next.js can move to any Node host.

Violates when: Choosing a vendor's proprietary visual page-builder because it's fast in week one. In year two the vendor raises prices and "the website" is hostage.

1.8 Vibe-coding requires guardrails.

"Talk to Claude, push code" is sustainable only with automated checks. Linting catches stylistic drift. Type-checking catches schema drift. Visual regression catches design drift. Without these, the agent's good days carry forward and bad days carry sideways.

Violates when: Three weeks of "just merging Claude's PRs." Week four: an unauthenticated Server Action ships, no one notices.

1.9 Security is a layer, not a sprinkle.

Threat-model before code, not after. Defense in depth. Treat every input as hostile. Cannot be retrofitted onto a codebase that was built without it. For AI code: agents have well-documented failure modes (slopsquatting, leaked secrets, mis-scoped RLS). The framework's job is to make these impossible to ship.

Violates when: A "quick" admin endpoint with no rate limit because "it's behind a login wall." Two weeks later the credential leaks via a log line.

1.10 The repo is the source of truth.

Not your laptop. Not your head. Every decision lives in an ADR. Every convention lives in AGENTS.md. Every spec lives in a spec.md. The test: could you onboard a new agent (or human) tomorrow from git clone + reading the repo?

Violates when: "Oh yeah, we always use this rate-limit value, I just remember it."

1.11 Optimize for the agent's failure modes, not its peak.

When Claude is brilliant, you don't need a framework. The design point is the worst Claude day. If your guardrails work then, they work every day. The framework should be paranoid, mechanical, and boring on the agent's behalf — that frees the agent to be creative without consequence.

Violates when: A Server Action that takes a userId from the request body. On a good day Claude reads the session. On a bad day it doesn't. The bad day ships.

1.12 Bias toward boring.

New tools are tax. Established patterns are leverage. Boring choices have more documentation, more agent training data, more migration paths. Postgres over the new graph DB. Markdown over the visual block editor. Server-rendered HTML over a SPA unless interactivity demands.

Violates when: Adopting a brand-new ORM with two months of GitHub history because a tweet said it was elegant.

1.13 Mobile is not a stretch goal.

If the public site doesn't work on phone, you have a broken site. If /admin doesn't work on phone, you cannot edit your own site from a coffee shop — which is the actual operational constraint. Treat any mobile breakage as a release-blocker.

Violates when: A drag-and-drop admin that's beautiful on desktop and unusable on iOS Safari. The site can't be edited from where it's actually edited.

1.14 Compound over time.

ADRs preserve why. Specs preserve what. AGENTS.md preserves how. The framework is good iff it pays back its own cost. The metric: time from "Cafe has an idea" to "live in production" should decrease as the framework matures.

Violates when: A repo with no ADRs, no spec history, no AGENTS.md. Every new feature re-debates settled questions.

Framework Rules — Layer 1 (Meta-Rules)
  1. Principles trump patterns. Violations require changing exactly one — never let both stand.
  2. Adding a principle is an ADR. The principle list is canonical and bounded.
  3. Cite the principle in the ADR. If none applies, the decision is probably an option, not a decision.
  4. Conway's Law applies. Build for one architect. Don't import patterns designed for ten-team orgs.
  5. When two principles collide, security wins.
Layer 2
The Specification Layer
The agent's blueprint. Without specs, the agent guesses. With specs, the agent reads.

Why specs matter for AI-augmented dev

A spec is not documentation of the code; it is the input to the code. The spec is executable, not descriptive. The agent reads it at the start of a turn and the human reviews at the end. The code in between is implementation detail.

For a solo + agent shop, the spec is the tribal knowledge. If it isn't written down, it doesn't exist.

The two dominant frameworks

GitHub Spec Kit: Constitution → Specify → Clarify → Plan → Tasks → Implement (CLI-driven, vendor-neutral).
Amazon Kiro: Requirements → Design → Tasks (IDE-integrated, EARS notation).

Recommendation for cafe655: Spec Kit folder layout + EARS acceptance criteria from Kiro, executed manually as Claude Code slash commands. Skip Spec Kit's "constitution" — that's AGENTS.md + ADRs.

## Summary
<!-- One sentence. If you can't, the spec is too big — split it. -->
Authenticate cafe655 admins via Supabase magic link with mandatory TOTP MFA,
restricted to a hard-coded email allow-list.

## Acceptance criteria (EARS)
- AC-1 (event): When an allow-listed email is submitted at /admin/login,
  the system shall send a Supabase magic link within 5 seconds.
- AC-2 (event): When a non-allow-listed email is submitted, the system
  shall return a 200 with a generic enumeration-safe message.
- AC-3 (event): When a valid magic link is followed, the system shall
  prompt for TOTP before establishing a session.
- AC-4 (state): While a session is unauthenticated or pending MFA, the
  system shall block all /admin/** routes.
- AC-5 (ubiquitous): The system shall use getUser() server-side, never
  getSession(). (Cross-ref: Layer 4, ADR-0007.)

## Constraints
- Security (Layer 4): AVAM enforced; CSP enforce mode; rate-limit at 5/min/IP.
- Performance (Layer 7): /admin/login TTI < 1.5s on cold edge cache.
- Accessibility (Layer 5): WCAG 2.2 AA on the login form.

EARS notation forms

  • Ubiquitous: "The system shall ..."
  • Event-driven: "When <trigger>, the system shall ..."
  • State-driven: "While <state>, the system shall ..."
  • Conditional: "If <condition>, then the system shall ..."
  • Optional: "Where <feature>, the system shall ..."

Framework Rules — Layer 2
  1. No code lands without a spec, except lightweight escape hatches.
  2. Specs are numbered, immutable in identity, and never deleted — only archived.
  3. Acceptance criteria are EARS notation, testable by construction.
  4. spec.md answers what and why only. How lives in plan.md.
  5. Every Layer-4 spec requires a paired ADR.
  6. Agent reads spec.md → plan.md → tasks.md, no further than necessary.
  7. Tasks are ticked off in the same PR that lands their code.
  8. Closed specs are archived with a closed-out summary in the same merge commit.
  9. A spec drifting from its implementation is a bug.
  10. notes.md is for the human, not the agent.

Layer 3 · highest leverage
The Agent Operating Manual
The standing instructions the agent reads on every turn. Brevity is load-bearing.

The cross-tool standard

AGENTS.md is the open Linux Foundation spec, supported by Claude Code, Codex, Cursor, Aider, Copilot, and 25+ tools across 60,000+ projects. CLAUDE.md is the Claude-specific extension. The dominant convention — used by vercel/next.js — is to symlink CLAUDE.md → AGENTS.md.

The agent reads the nearest AGENTS.md walking up from the working file. Sub-projects override the root.

Brevity is load-bearing

Every line of AGENTS.md is loaded into context on every turn. Frontier LLMs reliably follow ~150–200 instructions; Claude Code's system prompt already uses ~50. That leaves ~100–150 for everything else.

  • Target ≤300 lines, ideally under 60. Vercel's Next.js AGENTS.md at 446 lines is the outer bound for a framework-scale codebase.
  • Auto-generated AGENTS.md fails. Noise drowns signal.
  • Every line should be a rule, a fact, or a pointer. No prose explaining why — that's what ADRs are for.

PII hygiene

AGENTS.md and CLAUDE.md are committed. They will be read by every contributor, every agent, and every passerby if leaked. Never include real customer names/emails, API keys, service-role keys, DB connection strings, or internal-only URLs.

The mental test: would I be comfortable if this file were on the front page of Hacker News tomorrow?

  1. AGENTS.md is hand-curated. Auto-generators are forbidden.

  4. CLAUDE.md is either a symlink to AGENTS.md (preferred) or includes it via @AGENTS.md.
  5. The forbidden-patterns list is a flat bulleted list, each item naming its defended layer.
  6. No PII, no secrets, no real customer data, no internal-only URLs.
  7. Detail beyond root lives in docs/*.md and is loaded on demand.

  10. A stale rule in AGENTS.md is a bug; fix it in the same PR that breaks the rule.

4.2 The AVAM pattern — every Server Action, every time

AVAM — non-negotiable
Authenticate — const user = await getUser(). Return early if null.
Validate — Zod-parse all inputs. Reject on parse failure.
Authorize — check the user is allowed to do this thing to this resource.
Mutate — only after the above three pass.
'use server';
import { z } from 'zod';
import { getUser } from '@/lib/auth';

const Input = z.object({ pageId: z.string().uuid(), title: z.string().min(1).max(120) });

export async function updatePageTitle(formData: FormData) {
  const user = await getUser();                              // Authenticate
  if (!user) throw new Error('Unauthorized');
  const data = Input.parse(Object.fromEntries(formData));    // Validate
  await assertCanEditPage(user.id, data.pageId);             // Authorize
  return await db.pages.update({ id: data.pageId, title: data.title }); // Mutate
}

4.3 Supabase Row-Level Security (RLS)

Mental model: Postgres RLS is default-allow until you enable row level security. Once enabled with no policies it becomes default-deny. Service role key bypasses all RLS.

alter table notes enable row level security;

create policy "users read own notes"
  on notes for select
  to authenticated
  using ( (select auth.uid()) = user_id );

create policy "users insert own notes"
  on notes for insert
  to authenticated
  with check ( (select auth.uid()) = user_id );

Wrapping auth.uid() in (select ...) lets Postgres cache it per-statement.

4.8 Slopsquatting — supply chain

March 2025 study: ~20% of LLM-suggested packages don't exist. 43% reproducibly. Attackers monitor LLM hallucinations and pre-register the names with malicious payloads.

Pre-merge checklist for AI-authored PRs
  • Every new import resolved on npmjs.com and matches expected maintainer.
  • No new dangerouslySetInnerHTML, eval, new Function.
  • Every Server Action follows AVAM.
  • Every new table has RLS + at least one policy.
  • No service-role key reachable from Client Components.
  • No secrets in code; gitleaks passes.

Layer 5
The Design System
Encode every visual decision once. Tokens or it didn't happen.

The 5 priority-ordered principles

  1. Tokens or it didn't happen. Every visual value lives in design.tokens.json. No raw hex, no magic px, no inline style.
  2. Semantic over primitive at the call site. Components reference --color-fg-default, never --color-gray-12.
  3. Accessibility is the floor, not the ceiling. Contrast encoded in tokens; focus rings global; every spec has an Accessibility section.
  4. Boring on purpose. Surprise is a bug. Variety lives in content, not chrome.
  5. One way to do it. Two ways to make a primary button = one too many.

- One primary per view
- Verb-first labels ("Save changes", not "Changes")

Accessibility

  • Contrast targets baked into tokens. Steps 11/12 are defined by APCA Lc 60/Lc 90. The agent doesn't measure contrast — it picks the right semantic name.
  • Focus rings — global, never per-component. :focus-visible { outline: 2px solid var(--color-focus-ring); outline-offset: 2px; }
  • Skip link in PageShell, always.
  • Reduced motion is global via prefers-reduced-motion; motion-safe opt-in for animations.
  • aria-live="polite" for status updates; "assertive" for errors.
  • Never encode meaning in color alone. Errors get icon + "Error:" prefix + red.
  • Tools. axe-core via @axe-core/playwright on every PR. Lighthouse a11y gate at 95.

Voice & content

  • Title case rules: sentence case for everything except site title and proper nouns.
  • Sentence length target: under 25 words avg. Lint warns above 25, fails above 35.
  • Link text: always descriptive. Never "click here," "read more" alone.
  • Button labels: verb-first ("Save changes," "Delete project"). Two words ideal, four max. Never "OK."
  • Error messages: three parts — what happened, what to do, where to go for help.
  • Empty states (4 elements): headline, explanation, illustration or icon-32, single CTA.

Sub-site theming — concrete

  • cafe655 (default). Red accent, Geist Sans, dark theme.
  • barista-painting (commercial). Blue accent (blue.9 #1d6df0), Geist Sans, light theme.
  • ai-field-notes (knowledge). Orange accent (orange.9 #f76b15), JetBrains Mono headings.

Crucial property: every component file is identical across subsites. Only tokens change.

Framework Rules — Layer 5
  1. Tokens or it didn't happen. No raw hex, px, ms, font-size outside tokens/. Lint enforces.
  2. Semantic names at the call site. Components reference fg-default/bg-accent-bg/focus-ring.

  4. Focus rings are global, defined by :focus-visible. Custom rings forbidden.
  5. Reduced motion is global. Animations use motion-safe:.
  6. Sub-site variation lives in tokens, not in component code.
  7. No new color, font, breakpoint, or radius without an ADR.
  8. Every PR runs axe-core + Lighthouse a11y + visual regression.
  9. Empty / loading / error states are first-class — every data-bound page has all four states designed before merge.

The complete proxy.ts (production-ready)

// src/proxy.ts
import { NextResponse, type NextRequest } from 'next/server'
import { createMiddlewareClient } from '@/lib/auth/proxy-client'
import { getRedirect } from '@/lib/seo/redirects'

const SUBSITE_HOSTS: Record<string, string> = {
  'ai-field-notes.cafe655.com': 'ai-field-notes',
  'projects.cafe655.com':       'projects',
}

export async function proxy(request: NextRequest) {
  const url = request.nextUrl
  const host = request.headers.get('host') ?? ''
  const pathname = url.pathname

  // 1) Per-request CSP nonce
  const nonce = Buffer.from(crypto.randomUUID()).toString('base64')
  const cspHeader = [
    `default-src 'self'`,
    `script-src 'self' 'nonce-${nonce}' 'strict-dynamic'`,
    `style-src 'self' 'unsafe-inline'`,
    `img-src 'self' blob: data: https:`,
    `connect-src 'self' https://*.supabase.co`,
    `frame-ancestors 'none'`,
    `base-uri 'self'`,
    `form-action 'self'`,
  ].join('; ')

  // 2) Redirect map
  const redirect = getRedirect(pathname)
  if (redirect) {
    return NextResponse.redirect(new URL(redirect.to, request.url), redirect.code)
  }

  // 3) Subdomain routing
  const subsite = SUBSITE_HOSTS[host]
  if (subsite && !pathname.startsWith(`/${subsite}`)) {
    const rewritten = url.clone()
    rewritten.pathname = `/${subsite}${pathname === '/' ? '' : pathname}`
    return applySecurityHeaders(NextResponse.rewrite(rewritten), nonce, cspHeader)
  }

  // 4) Auth gate for /admin
  if (pathname.startsWith('/admin')) {
    const supabase = createMiddlewareClient(request)
    const { data: { user } } = await supabase.auth.getUser()
    if (!user) {
      const login = url.clone()
      login.pathname = '/login'
      login.searchParams.set('next', pathname)
      return NextResponse.redirect(login)
    }
  }

  return applySecurityHeaders(NextResponse.next(), nonce, cspHeader)
}

# ADR 0002 — Keystatic over Sanity

## Decision (Y-statement)
In the context of building a multi-subsite hub edited primarily by a solo
non-developer with AI assistance, facing the trade-off between fast iteration
and long-term content portability, we chose **Keystatic** over Sanity to
achieve **single-source-of-truth content (in repo, alongside code)**,
accepting the cost of less polished real-time multi-user collaboration.

## Status: accepted

## Consequences
- + Content lives in the repo. The agent can edit copy in the same turn it edits code.
- + Zero hosted dependency. No second SaaS bill.
- + Git is the version history.
- + Content is portable Markdown + JSON.
- − No real-time multi-user editing (not a current need).
- − Smaller community than Sanity.

## Alternatives considered
- Sanity (rejected: hosted, content lake separate from code)
- Payload (rejected for now: needs persistent Node server)
- Custom build (rejected: 3–6 months to ship)
- Notion as CMS (rejected: vendor lock-in, fragile API)

The supersession lifecycle

An ADR you rewrite when you change your mind is a lie. An ADR you supersede is a memory. The body of the old ADR is not edited — only its status field flips.

                                       │ six months later
                                       ▼
ADR 0024 (status: accepted) → "Use RLS pattern B"
                                       │ in the same PR
                                       ▼
ADR 0003 (status: superseded by 0024)
Framework Rules — Layer 11
  1. ADRs use AgDR-style headers — agent, model, trigger, timestamp required.
  2. Every architectural change ships with its ADR in the same PR.
  3. ADRs are numbered, immutable in identity, never deleted.
  4. Superseded ADRs have status field updated and a dated note appended; body never rewritten.
  5. Every ADR contains a Y-statement, Context, Alternatives considered, and Consequences (+ and −).
  6. AGENTS.md instructs the agent to read the most recent 5 ADRs before any architectural suggestion.
  7. Trivial decisions do not get ADRs.
  8. New dependencies always get ADRs. No exceptions.
  9. ADRs cross-reference the spec they came from (Layer 2) and AGENTS.md sections they constrain (Layer 3).
  10. The template.md and README.md in docs/decisions/ are kept current.

Master Framework Rules
The non-negotiables, all in one place.

Code & architecture

  • One root layout. 'use client' only on leaves. No cross-site imports. Tokens not values.
  • New routes follow (sites)/_template/ unless ADR'd otherwise.
  • Content lives outside app/; new static HTML in public/ is forbidden (one allowed exception per ADR).
  • Every content shape has a Zod schema in lib/schemas/.

Security (Layer 4 — non-negotiable)

  1. AVAM in every Server Action.
  2. RLS on every table from create table.
  3. getUser() / getClaims() server-side; never getSession().
  4. Service role key never in Client Components or NEXT_PUBLIC_*.
  5. Strict, nonced CSP; report-only 7 days then enforce.
  6. MFA mandatory for admin; admin = single hard-coded email.
  7. Rate limit auth, email, LLM, unauth-write endpoints.
  8. Verify every new npm dep; lockfile committed; npm ci --frozen-lockfile.
  9. Audit log every privileged action.
  10. 3-2-1 backups; PITR day customer data lands.
  11. Vercel Analytics + Plausible default; no GA4 unless paired with consent manager.
  12. LLM input is data, not instructions.

Workflow (Layer 9)

  • Branch protection on main: PR + CI green + linear history + include administrators.
  • No direct pushes to main, ever.
  • Squash and merge only.

  • Push Protection / Dependabot / secret scanning all on.
